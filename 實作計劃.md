# 誰是臥底 Web 遊戲 — 實作計劃

## 0. 已確認的關鍵決策

| # | 決策 | 說明 |
|---|------|------|
| 1 | PlayerId 機制 | `localStorage` 存 `{ [roomId]: uuid }` 的 map，每個房間獨立身分。進新房間會拿到新 UUID。 |
| 2 | 平票處理 | 平票候選人重新發言 → 全體存活玩家針對候選人重新投票 → 仍平票則迴圈，直到分出勝負 |
| 3 | 部署容器 | **單一 Docker container** 包 frontend static + backend + redis-server（Redis 跑純記憶體模式，不做持久化） |
| 4 | 部署平台 | Render 免費 Web Service（接受冷啟動 ~30 秒） |
| 5 | Host 角色 | 獨立角色：不發牌、不輪發言、不投票；可斷線重連；斷線期間遊戲正常進行 |
| 6 | 最少人數 | 至少 4 名玩家（不含 Host）才能開始 |
| 7 | 題庫 | 800 題靜態 JSON，不做耗盡 fallback |
| 8 | 重啟資料 | 重啟即清空 Redis，所有房間失效（派對遊戲可接受） |

- **前端**：React 18 + Vite + Tailwind CSS + socket.io-client + react-router

## 3. Redis 資料模型

### Key: `room:{roomId}` (Hash)
```
status           → "waiting" | "playing" | "voting" | "revoting" | "finished"
settings         → JSON: { spyCount, blankCount, civilianCount }
words            → JSON: { civilianWord, spyWord }
hostId           → uuid
round            → int
currentSpeakerId → uuid
spokenThisRound  → JSON array of uuid
usedWordsIds     → JSON array of int
tieCandidates    → JSON array of uuid（平票重投時使用）
createdAt        → timestamp
```
**TTL：6 小時**，遊戲結束後縮短為 10 分鐘。

### Key: `room:{roomId}:players` (Hash, field = playerId)
每個 field 存一個玩家 JSON：
```json
{ "playerId":"uuid", "name":"玩家一", "role":"civilian|spy|blank",
  "word":"計程車", "isAlive":true, "isOnline":true, "hasVotedFor":null }
```

### Key: `room:{roomId}:votes` (Hash)
`field = 被投玩家 playerId`, `value = 票數`
→ 用 `HINCRBY` 保證原子性。

### Key: `socket:{socketId}` (String)
value = `{roomId}:{playerId}`，方便斷線時快速反查。

## 4. 核心邏輯重點（會寫成服務層）

### 4.1 房號生成
4 碼大寫英數，`SCAN` 或 `EXISTS` 檢查避重；衝突重試最多 5 次。

### 4.2 PlayerId & 重連
- 前端 `storage.js`：`getPlayerId(roomId)`，無則生成 UUID 存入 map
- `join_room` event 帶 `{ roomId, playerId, name }`
- 後端檢查 `room:{roomId}:players` 是否已有該 playerId：
  - 有 → 更新 `isOnline=true`、綁定新 socketId、推送當前遊戲狀態
  - 無 → 視為新玩家（若房間 status=waiting 才允許加入）

### 4.3 Host 角色
- Host 在 `room:{roomId}` 的 `hostId` 欄位，**不進 players hash**
- Host 一樣用 localStorage UUID，可重連
- 遊戲邏輯完全無視 Host 的上線狀態

### 4.4 發言階段
- `nextSpeaker()`：從 `alive && !spokenThisRound && isOnline` 中 `Math.random()` 挑一個
- 若全員離線 → 等待
- `finish_speaking`：**嚴格校驗** `socket對應的playerId === currentSpeakerId`
- 全員發言完 → `status = voting`

### 4.5 投票階段（原子 + 防重複）
- `vote` event 帶 `targetPlayerId`
- 後端流程：
  1. 檢查玩家 `hasVotedFor` 是否已設定 → 已投則拒絕
  2. `HSET` 設定 `hasVotedFor`
  3. `HINCRBY room:{roomId}:votes {targetId} 1`
  4. 檢查是否所有存活玩家都投完 → 進入結算

### 4.6 平票重投流程
1. 計算最高票玩家清單
2. 若只有 1 人 → 淘汰、進勝負判定
3. 若 >=2 人平票：
   - `status = revoting`
   - `tieCandidates = [那幾個 playerId]`
   - 清空 `votes` hash
   - 清空所有玩家 `hasVotedFor`
   - 讓候選人依序重新發言（沿用發言階段邏輯，但 `spokenThisRound` 只算這幾個人）
   - 發言完 → 全員只能對 `tieCandidates` 投票
   - 迴圈直到單一最高票

### 4.7 勝負判定（每次淘汰/踢除後都要跑）
- `aliveSpies = 存活臥底數`
- `aliveNonSpies = 存活平民+白板數`
- **平民勝**：`aliveSpies === 0`
- **臥底勝**：`aliveSpies >= aliveNonSpies`
- 否則 → 進下一輪發言，清空 `spokenThisRound`、`votes`、`hasVotedFor`、`round++`

### 4.8 踢人邏輯
- 只有 Host 可觸發
- 被踢玩家直接從 players 移除（或標記 `isAlive=false` + `kicked=true`）
- 立即跑勝負判定
- 若被踢者正是 `currentSpeakerId` → 立即 `nextSpeaker()`

### 4.9 斷線處理
- Socket `disconnect` → 透過 `socket:{socketId}` 反查，將玩家 `isOnline=false` 廣播
- 遊戲不暫停；若輪到離線者發言/投票 → 卡住等他重連或 Host 踢除

## 5. WebSocket 事件總表

### Client → Server
| 事件 | Payload | 說明 |
|------|---------|------|
| `create_room` | `{ hostId }` | 建房，回傳 roomId |
| `join_room` | `{ roomId, playerId, name }` | 加入/重連 |
| `start_game` | `{ roomId, spyCount, blankCount }` | Host 專用 |
| `finish_speaking` | `{ roomId }` | 當前發言人專用 |
| `vote` | `{ roomId, targetPlayerId }` | 投票 |
| `kick_player` | `{ roomId, targetPlayerId }` | Host 專用 |

### Server → Client
| 事件 | 說明 |
|------|------|
| `room_updated` | 玩家列表、狀態變動廣播 |
| `role_assigned` | **私發**給單一玩家：身分與詞彙 |
| `speaker_changed` | 當前發言人更新 |
| `voting_started` | 進入投票階段（帶候選清單） |
| `revote_started` | 平票重投（帶候選清單） |
| `player_eliminated` | 淘汰公告 |
| `game_over` | 勝負結果（含所有人身分揭露） |
| `error` | 統一錯誤事件 |

## 6. Dockerfile 策略（單一容器）

```dockerfile
# Stage 1: 建置前端
FROM node:20-alpine AS frontend-build
WORKDIR /app/frontend
COPY frontend/package*.json ./
RUN npm ci
COPY frontend/ ./
RUN npm run build

# Stage 2: 建置後端依賴
FROM node:20-alpine AS backend-build
WORKDIR /app/backend
COPY backend/package*.json ./
RUN npm ci --omit=dev
COPY backend/ ./

# Stage 3: 最終映像
FROM node:20-alpine
RUN apk add --no-cache redis
WORKDIR /app
COPY --from=backend-build /app/backend ./backend
COPY --from=frontend-build /app/frontend/dist ./backend/public
COPY start.sh ./
RUN chmod +x start.sh
ENV NODE_ENV=production
ENV PORT=10000
EXPOSE 10000
CMD ["./start.sh"]
```

**`start.sh`**：
```sh
#!/bin/sh
redis-server --save "" --appendonly no --daemonize yes --bind 127.0.0.1
node backend/src/server.js
```

後端 `server.js` 用 `express.static('public')` 同時服務前端 build，**前後端共用同一個 port**，省去 CORS 問題。

## 9. 已知風險與接受的取捨

| 風險 | 處理 |
|------|------|
| Render 冷啟動 ~30s | 接受，派對遊戲可容忍 |
| 重啟清空 Redis | 接受，房間本為短暫狀態 |
| 單容器無水平擴展 | 派對遊戲量級不需要 |
| 全員同時斷線 → 房間卡住 | 靠 6 小時 TTL 自動清理 |
| 題庫耗盡 | 不處理（800 題足夠） |
